# T-005 — Collapse matching/ports adapter plumbing (Option 1)

**Date:** 2026-04-21
**Branch:** develop
**Status:** Complete. Build + vet + `make lint` (touched packages) + `make test` (unit, 15,232 tests passing on T-005 surface; 1 pre-existing WIP failure in `exception/adapters/http` and 3 build errors are unrelated — see "Baseline contamination" below).

## Deviation from task spec

T-005 as originally written (`docs/pre-dev/simplify-2026-04-21/tasks.md`) asked
to delete all 7 matching ports + the 380-line `MatchingConfigurationProvider`
facade and inject concrete configuration / ingestion repositories directly
into the matching UseCase. After surveying the ground I stopped and reported
three reasons that path was infeasible:

1. **`cross-context-matching` depguard rule blocks the concrete injection
   path.** The rule denies `internal/configuration` wholesale (line 373) and
   denies `internal/ingestion/adapters` explicitly (line 386). T-003's carve-out
   opened `ingestion/domain/{entities, value_objects}` only, NOT the adapter
   path. Configuration has no carve-out at all.
2. **`service-no-adapters` blocks the in-context `ExceptionCreator`
   collapse.** Matching services cannot import `matching/adapters/postgres`
   either (line 584). So even the within-context port K-07e could not be
   collapsed to a concrete pointer.
3. **The facade does real translation work.** `ReconciliationSource →
   SourceInfo`, `ReconciliationContext → ReconciliationContextInfo`, source
   type string-cast mapping, cursor-pagination aggregation across multiple
   repo pages — none of that is ceremony. Deleting it means either pushing
   translation into the UseCase (pollutes the service layer) or deleting the
   matching-specific DTOs entirely (cross-context leak). Neither is in scope
   for a "simplification" pass.

The simplify report's K-07a-g premise was overconfident. T-003's own handoff
(`docs/handoffs/simplify/T-003.md`, lines 37-45) acknowledged this and named
interface-relocation (T-016 style) as the "more likely" alternative — but
even that is a larger refactor than T-005 warrants. The user confirmed
Option 1 (adapter plumbing collapse only) as the correct scope.

## Option 1 deliverable

The collapse target is the 4 `*ProviderAdapter` wrappers inside
`shared/adapters/cross/matching_adapters.go`. They existed solely to provide
typed satisfaction of the 4 matching port interfaces while delegating to
`MatchingConfigurationProvider`'s private methods — textbook ceremony over
an already-correct concrete type.

### What was collapsed

`MatchingConfigurationProvider` now implements 3 of the 4 matching port
interfaces directly:

| Port interface | Receiver method on provider |
|---|---|
| `matchingPorts.ContextProvider` | `FindByID(ctx, tenantID, contextID)` |
| `matchingPorts.SourceProvider` | `FindByContextID(ctx, contextID) → []*SourceInfo` |
| `matchingPorts.MatchRuleProvider` | `ListByContextID(ctx, contextID)` |

Compile-time assertions (`var _ matchingPorts.X = (*MatchingConfigurationProvider)(nil)`)
replaced the per-adapter satisfaction checks.

### Why 3 of 4, not 4 of 4

Go disallows two methods with the same name on the same receiver.
`SourceProvider.FindByContextID` returns `[]*SourceInfo`;
`FeeRuleProvider.FindByContextID` returns `[]*fee.FeeRule`. Same name,
same parameter list, different return type. Cannot coexist on
`*MatchingConfigurationProvider`.

**Minimum-surgery resolution:** keep `FeeRuleProviderAdapter` as a
thin view type (2-field struct; single method) reached via
`provider.FeeRuleProvider()`. The other three ceremony-only adapter
types are gone.

| Symbol | Reason |

| `MatchingConfigurationProvider` | Real translation work (`ReconciliationContext → ReconciliationContextInfo`, `ReconciliationSource → SourceInfo`, cursor pagination) — not ceremony |
