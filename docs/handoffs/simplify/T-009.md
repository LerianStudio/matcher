This collapses 2 UseCase methods across 2 contexts. ## Why the Remaining 21 Methods Were Deferred

Five categories of blocker surfaced that were not visible in the task spec or simplify report:

### 1. Verifier cascade (configuration `GetContext`, `GetSource`, `GetMatchRule`)

`internal/configuration/adapters/http/verifier.go` is a `libHTTP.TenantOwnershipVerifier` used by every context-scoped route across the API surface. It calls `queryUseCase.GetContext(ctx, contextID)`. Collapsing `GetContext` requires either:

- Rewiring `NewTenantOwnershipVerifier` to take a repo instead of a UseCase (cascades to every call site + tests), or
- Leaving `GetContext` on the UseCase purely to feed the verifier (contradicts the task goal).

The same pattern applies to `GetSource` (called from `ensureSourceAccess` helper used by multiple handlers). This is exactly the "subtle work" risk flagged by R-T009-2 in the task spec — the body is span-only but the consumer is load-bearing.

### 2. Handler doesn't currently inject the repo

For every handler that would lose a UseCase method (configuration, matching, reporting, ingestion, discovery), the `Handler` struct holds `query *query.UseCase` but NOT the underlying repo. Each collapse requires:

- Adding a `repo` field to the handler struct.
- Updating the constructor to accept the repo.
- Updating bootstrap to pass the repo.
- Updating every test fixture that constructs a handler (many).

For reporting alone: 10 methods × touching `Handlers`, `NewHandlers`, `handlers_export_job.go`, `handlers_coverage_test.go`, `handlers_export_job_test.go`, etc. Each is ~1 min of mechanical work but the aggregate reaches 4–6 hours — beyond the T-009 budget on top of the exception/governance work already landed.

### 3. Nil-result → sentinel (`GetException`, `GetDispute`, `GetConnection`, `GetExtraction`, `GetByID`)

These methods look span-only but include `if result == nil { return ErrXNotFound }`. Collapsing forces the handler to duplicate that check. Small, but it's logic — not a pure span wrap. Marked as "KEEP" per the task's own rule: "Do NOT collapse UseCases with real logic".

### 4. Conditional-dispatch methods (`ListSources`, `ListMatchRules`, `PreviewFile`)

Body inspects a nullable filter (e.g. `sourceType *value_objects.SourceType`) and picks between two repo methods. This is real routing logic. KEEP.

### 5. sql.ErrNoRows log-filter (`GetFieldMap`, `GetFieldMapBySource`)

Body wraps the repo call but suppresses `HandleSpanError` / `logger.Error` when the error is `sql.ErrNoRows` (because the handler translates 404 and noisy error logs would flood production). Collapsing requires moving this filter into the handler — small logic, but logic. KEEP (or accept the extra handler code as a follow-up).

## What Unblocks Completion

Two clean paths forward, depending on Fred's appetite:

1. **Finish the mechanical 18 (the `*Count*`, `*List*`, `GetMatch*`, etc.)**: These are genuinely span-only with no verifier coupling. Estimate 2–3 hours of careful handler plumbing + test updates. Should be done as a sibling task with its own scope (T-009b), not squeezed into T-009.
2. **Address the verifier cascade (the `GetContext`/`GetSource`/`GetMatchRule` trio)**: Convert verifiers to take repos. This changes the shape of `NewTenantOwnershipVerifier` across 4 contexts (configuration, matching, ingestion, reporting). Non-trivial but unlocks ~6 more method collapses. Worth a separate decision.

The simplify report's "15–20" estimate assumed handlers already had repos plumbed and ignored the verifier cascade. Both assumptions were wrong.
