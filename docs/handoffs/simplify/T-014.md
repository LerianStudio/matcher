# T-014: Remove Postgres UUID-as-string round-trip

**Date:** 2026-04-21
**Status:** Complete
**Mode:** Direct implementation

## Summary

Eliminated `uuid.Parse(...)` + `.String()` round-trips in PostgreSQL adapter model
structs across all bounded contexts. PG model UUID fields now use native
`uuid.UUID` / `uuid.NullUUID` types instead of `string` / `sql.NullString`.
The pgx/v5 stdlib driver handles scalar UUID scan/value via `google/uuid`'s
built-in `sql.Scanner` and `driver.Valuer` implementations — no extension
required. Array-of-UUID for `squirrel.Eq` IN-clauses continues to use `[]string`
to sidestep pgx text-mode array encoding (documented constraint per MEMORY.md
Feb 2026 finding).

## Pre-flight: pgx driver native UUID support

**Verified working without `pgxuuid` extension.** The codebase uses
`github.com/jackc/pgx/v5 v5.9.2` via `pgx/v5/stdlib` (database/sql compatibility
layer). Scalar `uuid.UUID` fields work transparently because:

- `google/uuid.UUID.Scan(src interface{})` accepts string and `[]byte`.
- `google/uuid.UUID.Value()` returns the canonical string representation.

Only the `[]uuid.UUID` + `squirrel.Eq{"id": ids}` path requires pgx array
extension work — that pattern is preserved with `[]string` boundary conversion
via `id.String()` in the two places where it matters:

- `ingestion/adapters/postgres/transaction/transaction.postgresql.go` (insertedIDs batch)
- `matching/adapters/postgres/fee_schedule/fee_schedule.postgresql.go` (scheduleIDs)

### Test fixture/assertion updates

Every adapter's `*_test.go`, `*_sqlmock_test.go`, and helper files were updated
to:
- Pass `uuid.UUID` values instead of string-constructed UUIDs in model struct
  literals.
- Replace `require.Equal(t, x.String(), model.Foo)` with
  `require.Equal(t, x, model.Foo)`.
- Replace `require.Contains(t, err.Error(), "parsing ID")` with
  `require.Contains(t, err.Error(), "invalid UUID")` (scan-level error).
- Extend mockScanner helpers in match_group and dispute to handle
  `*uuid.UUID` / `*uuid.NullUUID` destinations.
- Delete obsolete `TestToEntity_InvalidID` / `InvalidContextID` tests whose
  only purpose was exercising the now-nonexistent `uuid.Parse` path on the
  model field. The scan-level UUID rejection is covered by the existing
  `TestScan*_InvalidUUID` tests.

## Semantic changes

**None in happy-path behavior.** Only observable differences:

- Error messages for invalid UUID data now come from
  `sql.Rows.Scan`/`uuid.UUID.Scan` with substring `"invalid UUID"` instead of
  `"parsing ID"`. Tests updated accordingly.
- `GetContextIDBySourceID` behavior on empty-string DB value: previously
  returned `"parse context id"` error (via `uuid.Parse("")` which errors);
  now returns `uuid.Nil, nil` (via `uuid.UUID.Scan("")` which silently
  succeeds as per google/uuid contract). The defensive test
  `TestGetContextIDBySourceID_EmptyContextID` was deleted — the
  DB column is `NOT NULL UUID`, so empty strings are a pathological case
  that cannot occur in production. The existing
  `TestGetContextIDBySourceID_NilUUIDContextID` (which passes
  `uuid.Nil.String()`) still covers the "DB returned zero UUID" case.

## Preserved constraints

- Array-of-UUID → `squirrel.Eq` still uses `[]string` with
  `.String()` at boundary. Not changed; pgx array encoding remains fragile
  per MEMORY.md Feb 2026 finding.
- Domain entities (`shared.Transaction`, `matchingEntities.MatchGroup`,
  etc.) already used `uuid.UUID`. No entity changes required.
- Migrations unchanged (columns were already `UUID` per CLAUDE.md).
- JSONB handling untouched.
- No dependency changes; pgx v5.9.2 retained.
