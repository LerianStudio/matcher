# T-013 — Evaluate repository-over-SQL interfaces (per-context, burden-of-proof)

**Date:** 2026-04-21
**Scope:** Evaluation of all 24 repo interfaces under `internal/*/domain/repositories/`. Three-option framework applied (DELETE / COLLAPSE-ADAPTER / RELOCATE / KEEP).
**Status:** Evaluation complete. **0 collapses. 24 KEEPS.** No code changes. Baseline lint + tests still green.

## TL;DR

The simplify report's R-01 projection (~3,000 LOC win by collapsing 23 repository interfaces) does not survive contact with the codebase. **Every repository interface earns its keep** through one or more of:

- **Generated mockgen mocks** with active test consumers (15 of 24 interfaces)
- **Handcrafted stub structs** with explicit `var _ repositories.X = (*stub)(nil)` compile-time assertions (most of the rest)
- **Cross-context wiring through depguard-safe boundaries** — bootstrap injects concrete repositories from one context into another context's UseCases via the interface seam; collapsing forces the consumer to import the producer's `adapters/postgres/...` directly, which depguard `cross-context-*` rules block.

Following the established precedent of T-016 (relocate, don't delete), T-005 (collapse adapter plumbing only), T-004 (collapse 2 of 15 — KEEP 13), R-03 (KEEP all 4), R-04 (KEEP), the honest verdict is: **R-01's projection of ~3,000 LOC is a paper estimate built on a constraint model the codebase does not satisfy.** The interfaces are not "pretending to be storage-abstract." They are **mockability seams + depguard contracts**. Both are real, both are load-bearing.

This is the sixth task in the simplify wave to converge on the same finding: the rubric's default verdict ("delete single-impl interface") is systematically too aggressive for this codebase's testing convention (sqlmock + handcrafted stubs at the interface seam) and architectural posture (strict cross-context isolation enforced by depguard).

## Methodology

For each of the 24 repository interfaces:

1. **Mock infrastructure inventory** — count generated mockgen mocks (`internal/*/domain/repositories/mocks/*_mock.go`), find handcrafted stub structs in `_test.go` files, and locate compile-time interface assertions (`var _ repositories.X = (*Y)(nil)`).
2. **Cross-context wiring** — inspect `internal/bootstrap/init.go` to identify which interfaces cross context boundaries via dependency injection. Cross-context use makes the interface depguard-load-bearing.
3. **Adapter plumbing check** — search for adapter structs that wrap a concrete repository solely to implement the interface (Option 1 candidates). Found none.
4. **Relocation candidates** — interfaces whose semantic owner is a different context than where they live (Option 2). T-016 already handled the only such case (`AuditLogRepository`).
5. **Cost-of-collapse estimate** — multiply test-stub count by ~5 LOC rewrite-per-stub. If any single interface's collapse would require >10 test rewrites, it fails the simplify-by-rewriting-tests bar (this hits all interfaces with active mocks/stubs).

## Why every collapse fails the burden-of-proof test

The simplify report's R-01 made three implicit assumptions, each of which is false here:

1. **"Every interface has exactly one impl, so it's not actually polymorphic."** True about production impls. False about test impls. **15 of 24 interfaces have generated mockgen mocks. The remaining 9 have handcrafted stubs.** The interfaces are polymorphic at test time, by design — the project's testing convention is sqlmock at the adapter level + interface stubs at the service level. Both are explicit choices recorded throughout the codebase via `var _ repositories.X = (*stub)(nil)` compile-time assertions.

2. **"Interfaces leak SQL shape (`*sql.Tx`) so they aren't truly abstract."** Correct observation, wrong conclusion. The interfaces are not pretending to abstract over storage technology — they are abstracting over **identity** (test stub vs. production repo) and **transaction composition** (caller-owned tx vs. self-managed). The `WithTx` variants are deliberate ergonomic seams for transaction composition (T-013's own scope explicitly says "Keep `WithTx` methods (real ergonomic need)"). The `*sql.Tx` parameter is a feature of the abstraction, not a betrayal of it.

3. **"Net ~3,000 LOC savings."** This estimate counted the LOC of interface declarations + mock files but did not count the LOC of test rewrites or the cost of widening depguard cross-context rules. A realistic estimate for a full collapse:
   - Delete interface declarations: ~−400 LOC
   - Delete mockgen mock files: ~−2,000 LOC (these are large generated files)
   - **Rewrite handcrafted stubs against concrete + sqlmock: +800–1,200 LOC**
   - **Widen depguard `cross-context-*` rules to allow concrete cross-context imports: ~+30 LOC of YAML, but this constitutes architectural retreat — the linter currently enforces hex layering, and weakening it to enable a refactor is the wrong vector.**
   - Net: ~−1,600 LOC at best, with depguard regression. The "~3,000 LOC" figure inflates the win and ignores the cost.

   And that's before counting the regression risk of touching ~250 import sites without behavior tests for the modified call paths.

## What would actually be collapsable, and isn't here

Genuine Option-1 (COLLAPSE-ADAPTER) candidates would have an adapter struct that wraps a concrete repo to implement the interface that the concrete already satisfies. **None of the 24 repository interfaces have such adapter wrappers.** Bootstrap passes concretes directly to interface-typed setters. The interface satisfaction is implicit and immediate.

Genuine Option-2 (RELOCATE) candidates would have an interface in the wrong package — e.g., a governance-owned interface living in `shared/ports`. T-016 already handled that case (`AuditLogRepository`). No remaining interfaces exhibit this.

Genuine DELETE candidates would have **zero mock consumers, zero handcrafted stubs, zero compile-time assertions, AND zero cross-context use**. **None of the 24 interfaces meet all four conditions.** The closest miss is `FeeRuleRepository` (zero mockgen consumers) — but it has three handcrafted stubs with `var _` assertions and is wired cross-context into matching.
