# T-018 — Split `bootstrap/init.go` by bounded context

**Status:** Complete
**Date:** 2026-04-21
**Branch:** develop (uncommitted)
**Scope:** `internal/bootstrap/`

## Goal

Reduce `internal/bootstrap/init.go` monolithicity by extracting each
bounded-context `initXxxModule` function into its own `init_{context}.go`
file, following the precedent set by `init_discovery.go`,
`init_fetcher_bridge.go`, `init_startup_probe.go`, and
`init_status_archival.go`.

Structural refactor only — no behaviour change, no signature change, no
call-site reordering.

## What did NOT move

- `initAuthModule` — does not exist. Auth wiring in bootstrap is split across
  `buildTenantExtractor`, `createAuthClient`, and `initializeAuthBoundaryLogger`,
  each a small shared helper; kept in `init.go` with other composition-root
  helpers.
- `initOutboxModule` — does not exist. Outbox handler wiring lives in
  `outbox_wiring.go`; the dispatcher itself is constructed inline in
  `initModulesAndMessaging` as a fifteen-line step. Extracting it would have
  required inventing new structure; task mandate is no behaviour change.
- `initGovernanceAdditionalRoutes` — does not exist. Only
  `initGovernanceModule` existed; everything it does moved to
  `init_governance.go`.
- `InitServersWithOptions`, `initModulesAndMessaging`,
  `initSharedRepositories`, `initEventPublishers`, `initCleanupMetrics`,
  telemetry/health/Postgres/Redis/RabbitMQ wiring, tenant-manager plumbing,
  all `resolve*` / `configured*` runtime-settings helpers, `cleanup*`
  functions, schema/Redis mode helpers — all remain in `init.go` because
  they are shared plumbing, not per-context.

## init.go still > 500 LOC — rationale

Aspirational target was < 500 LOC per bootstrap file. Post-split `init.go`
is 2 781 LOC. This is expected and correct. The residual content is:

- `InitServersWithOptions` orchestration (~370 LOC) — the composition root.
- Infrastructure helpers: telemetry, healthchecks, tenant manager,
  connection wiring, cleanup metrics, event-publisher factories, runtime
  settings resolver — ~1 700 LOC. Shared by every bounded context.
- `initModulesAndMessaging` (~260 LOC) — the per-context init dispatcher
  that owns ordering and shared dependencies. Task spec explicitly says the
  call-site order stays put.
- `initSharedRepositories`, `initEventPublishers`, runtime infra setup
  (~400 LOC) — not per-bounded-context.

Further reduction would require a separate task with its own scope
(extracting infra setup into `init_infra.go`, tenant manager wiring into
`init_tenant_manager.go`, etc.). Out of T-018's scope.

`init_exception.go` (297 LOC) is the largest new file. It clusters six
items (`initExceptionModule` + two types + three helpers) that are
tightly coupled; splitting further would create artificial boundaries.
