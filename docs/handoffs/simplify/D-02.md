# D-02: DTO snake_case → camelCase rename

**Date:** 2026-04-21
**Scope:** 4 HTTP DTOs across governance + exception contexts, one e2e client mirror, two JSON-shape tests, regenerated Swagger.
**Cross-repo coordination:** [lerianstudio/product-console#317](https://github.com/lerianstudio/product-console/issues/317)

## Summary

Four DTOs drifted from the project-wide `camelCase` JSON convention to `snake_case`. Per D-02 resolution in `docs/pre-dev/simplify-2026-04-21/tasks.md`, the drift is corrected atomically in Matcher with parallel coordination in `product-console` (the sole known consumer). Go struct field names unchanged; only `json:"..."` tag values flipped. No behavior change.

## Mirror updates (e2e client, non-public)

Updated `tests/e2e/client/types.go` definitions of `BulkResolveRequest`, `BulkFailure`, and `ActorMappingResponse` to match server tags. These are test-helper types that (de)serialize the same wire format — if the server changes, the client must follow or e2e tests break silently at the unmarshal boundary.

## Out-of-scope drift flagged (not fixed)

Task spec explicitly listed 4 DTOs. During implementation I found **related** snake_case usage in adjacent DTOs that was intentionally left alone:

| DTO | File | Remaining snake_case | Why unchanged |
|---|---|---|---|
| `UpsertActorMappingRequest` | `internal/governance/adapters/http/dto/actor_mapping.go` | `display_name` | Paired *request* for `ActorMappingResponse`. Now asymmetric: server returns `displayName` but accepts `display_name`. Frontend has to map both ways until this is fixed. |
| `BulkAssignRequest` | `internal/exception/adapters/http/dto/bulk_requests.go` | `exception_ids` | Sibling bulk-op DTO; same argument pattern as `BulkResolveRequest`. |
| `BulkDispatchRequest` | `internal/exception/adapters/http/dto/bulk_requests.go` | `exception_ids`, `target_system` | Same as above. |
| `ErrAtLeastOneFieldRequired` error text | `internal/governance/adapters/http/handlers_actor_mapping.go:30` | Error message references `display_name or email` | Runtime error string, not a JSON tag. User-facing text now drifts from the actual JSON key (`displayName`). |
| `UpsertActorMappingRequest` doc comment | `internal/governance/adapters/http/dto/actor_mapping.go:10` | `display_name or email must be provided` | Same rationale as above — not a tag. |

**Recommendation:** Fold these into a follow-up task or expand D-02 scope. The asymmetry on `ActorMappingResponse`/`UpsertActorMappingRequest` is particularly ugly — same endpoint, same field, two casings. Fred's call on whether to roll forward now or ship as-is and schedule the follow-up.

## Cross-repo coordination

Frontend must update the TypeScript types that mirror these DTOs — the issue enumerates each renamed field. Ship order: this Matcher change + the product-console companion PR land together, or behind a short-lived backend toggle. Recommend landing together since Matcher is pre-public and there are no other deployed consumers.
