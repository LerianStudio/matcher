## Decision

**KEEP.** The adapter does real translation work and relocating it either leaks configuration business rules into ingestion or requires widening depguard beyond what T-003's D-01 carve-out permits.

## Why this reverses the simplify-report premise

The simplify report (`docs/dev-simplify/simplify-report-2026-04-21.md` line 92) grouped R-04 with R-03 under D-01: "Tied to R-03 decision." The implicit assumption was that resolving D-01 (fetcher-bridge depguard carve-out) would also unlock R-04.

**That assumption was wrong.** D-01's resolution carved out `discovery → ingestion UseCase` for the fetcher bridge intake path. R-04's collapse would need a different carve-out: `ingestion → configuration/domain/repositories`. The two carve-outs are unrelated — sharing the same depguard rule family doesn't make them the same decision.

Reading the task handoff for T-004 (`docs/handoffs/simplify/T-004.md` line 24) confirms this adapter was explicitly kept three days ago for the same reason — real translation, not ceremony. Nothing has changed in the interim to alter that call.

## What the adapter actually does

```go
func (adapter *AutoMatchContextProviderAdapter) IsAutoMatchEnabled(
    ctx context.Context, contextID uuid.UUID,
) (bool, error) {
    // nil guards…
    ctxEntity, err := adapter.repo.FindByID(ctx, contextID)
    if errors.Is(err, sql.ErrNoRows) { return false, nil }   // translate not-found → false
    if err != nil { return false, fmt.Errorf("check auto-match enabled: %w", err) }
    if ctxEntity == nil { return false, nil }
    return ctxEntity.AutoMatchOnUpload && ctxEntity.IsActive(), nil  // business rule
}
```

Three pieces of real work:

1. **Not-found translation:** `sql.ErrNoRows` → `(false, nil)`. Without this, every ingestion upload into a deleted context would emit a spurious warn-log.
2. **Nil-entity translation:** defensive; repo contract allows `(nil, nil)` in some backends.
3. **Business rule composition:** `AutoMatchOnUpload && IsActive()`. This is the answer to ingestion's question "should I auto-match?" — not a data fetch.

## Why inlining loses

Two plausible inline shapes, both worse than keeping:

### Inline shape A: widen the port to return the entity

```go
type ContextProvider interface {
    FindByID(ctx context.Context, id uuid.UUID) (*configEntities.ReconciliationContext, error)
}
```

`configRepositories.ContextRepository` structurally satisfies this — bootstrap wires `repos.configContext` directly. Adapter deleted. Ingestion inlines:

```go
ctxEntity, err := uc.contextProvider.FindByID(ctx, contextID)
if errors.Is(err, sql.ErrNoRows) || ctxEntity == nil { return }
if !ctxEntity.AutoMatchOnUpload || !ctxEntity.IsActive() { return }
uc.matchTrigger.TriggerMatchForContext(ctx, tenantID, contextID)
```

**Cost:** ingestion now encodes the business rule *"auto-match is enabled iff `AutoMatchOnUpload` AND context is in Active status."* That rule belongs to configuration. Confirmed by grep: ingestion today has zero references to `IsActive()` or `ContextStatusActive`. Inlining creates that knowledge. Depguard allows the import (T-003 carve-out for `configuration/domain/entities`) but **allowing ≠ wanting**. The bounded-context boundary exists at the semantic layer, not just the import layer.

### Inline shape B: inject `ContextRepository` directly

Would require depguard carve-out for `ingestion → configuration/domain/repositories` (currently denied at `.golangci.yml:353`). Task instruction explicitly forbids widening beyond T-003's D-01. Rejected.

### Why not the T-005 precedent?

T-005 inlined matching's `MatchingConfigurationProvider` port methods directly onto the provider type. That worked because the provider already held the translation logic; the adapters were pure delegation shells. Here the inverse holds: the adapter *is* the translation logic, and the underlying `ContextRepository` is a generic CRUD interface that doesn't know about auto-match. The T-005 pattern doesn't apply.

## The ceremony test

The simplify report characterised this as "a cross-context shim over a 3-line body" (line 92). True LOC count, but LOC is the wrong metric. The right metric is: **does the code at the boundary do translation work that belongs somewhere?**

| Location | Natural fit for `AutoMatchOnUpload && IsActive()`? |
|---|---|
| `shared/adapters/cross/` (current) | Yes — bridge layer translating configuration domain → ingestion-facing boolean |
| `ingestion/services/command/` | No — encodes configuration business rule in ingestion |
| `configuration/domain/entities/` | Would need a new method `IsAutoMatchEligible()` — defensible, but a larger refactor than R-04 warrants and duplicates existing `IsActive()` + `AutoMatchOnUpload` readability |

The adapter is at the right boundary. Keep it.
