# T-016 — Collapse `AuditLogRepository` port (R-09)

**Date:** 2026-04-21
**Scope:** Option 2 (revised) — relocate the canonical interface into
governance/domain/repositories; do NOT convert consumers to concrete pointers.
**Status:** Complete. Build + vet (all tags) + lint (on touched packages) +
`make test` (unit, 15,580 tests) + `make test-int` (integration, 689 tests,
including hash chain suite) all green.

## Why Option 2

The literal R-09 reading ("replace the port with a concrete pointer") collided
with ~70 test call sites that currently implement the interface via stubs or
mocks. Converting those to real `*governancePostgres.Repository` instances
backed by sqlmock would have inflated test LOC by hundreds and turned a batch
of matching/exception unit tests into integration-style scaffolding — net
LOC-positive and net-fragility-positive, which is the opposite of the
simplify goal.

R-09's actual value is to give the audit repository contract a single
canonical home instead of bouncing between `shared/ports` and
`governance/domain/repositories` via type alias. Moving the interface
definition to governance delivers that while leaving the existing mocks +
stubs alone.

## Interface location

Canonical home: `internal/governance/domain/repositories/audit_repository.go`.

Converted from a `type AuditLogRepository = sharedPorts.AuditLogRepository`
alias to a real interface definition (5 methods, 48 lines of signatures +
doc comments). `Tx` is also a real alias to `*sql.Tx` in the same file, so
callers importing only `governance/domain/repositories` do not need
`database/sql`.

Design note: `governance/adapters/postgres/audit_log.postgresql.go` — the
sole production implementation — was considered as the interface's home
(co-locate with concrete). Rejected because hex layering expects repo
interfaces in `domain/repositories`, not `adapters/postgres`. Governance is
the one context where this convention was already broken by the alias
forwarding; T-016 fixes that.

## Depguard carve-out adjustments

Two rules in `.golangci.yml` were narrowed, not widened. The existing broad
deny on `github.com/LerianStudio/matcher/internal/governance/domain` was
split into three explicit sub-package denies (`entities`, `errors`,
`hashchain`), leaving `governance/domain/repositories` implicitly permitted.
This is the minimum change required to let matching + exception import the
newly-located `AuditLogRepository` interface.

**`cross-context-matching` and `cross-context-exception`** — identical change:

```diff
-            - pkg: "github.com/LerianStudio/matcher/internal/governance/domain"
-              desc: "Cross-context violation: Matching cannot import Governance domain. Use shared/ports or shared kernel."
+            - pkg: "github.com/LerianStudio/matcher/internal/governance/domain/entities"
+              desc: "Cross-context violation: Matching cannot import Governance entities. Use shared kernel (internal/shared/domain) for cross-context types."
+            - pkg: "github.com/LerianStudio/matcher/internal/governance/domain/errors"
+              desc: "Cross-context violation: Matching cannot import Governance errors. Governance sentinels are internal to that context."
+            - pkg: "github.com/LerianStudio/matcher/internal/governance/domain/hashchain"
+              desc: "Cross-context violation: Matching cannot import Governance hashchain logic. Hash chain is a governance implementation detail."
```

**Carve-out #4 comments** in both rules were rewritten to describe Option 2
reality (interface-in-governance, bootstrap-wired concrete) instead of the
Option 1 fantasy (direct concrete access from consumers). The permission
surface for `governance/adapters/postgres` remains unchanged — it was
permitted before (implicitly, by listing only the two forbidden sub-paths
`actor_mapping` and `archive_metadata`) and is still permitted now. The
carve-out does not widen read surface: matching + exception could already
call `GetByID`, `ListByEntity`, `List` through the port; they can still do
so through the interface.

`service-no-adapters` was **not** modified. Its existing narrowing (which
permitted `governance/adapters/postgres/audit_log` by omission) remains
valid for bootstrap-side wiring, though no service-layer code under Option 2
actually imports that adapter path — the services import the interface from
`governance/domain/repositories` instead. The carve-out is now defensive
rather than load-bearing; leaving it in place costs nothing and guards
against future drift.

No new rules added. No rules widened. The minimum-change principle holds.
