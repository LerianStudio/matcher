# T-015: Collapse ObjectStorageClient + ExtractionJobPoller + SchemaCache with atomic-pointer swap

**Date:** 2026-04-22
**Status:** Complete
**Mode:** Direct implementation

## Summary

Replaced three ports (`ObjectStorageClient`, `ExtractionJobPoller`, `SchemaCache`)
and their bootstrap-level `dynamic_*` wrappers with concrete types that use
`atomic.Pointer[T]` for hot-reload. The ports existed to model hot-reload
as "substitutable implementations" but had exactly one production backend
each; the `dynamic_*` wrappers were the actual state-swap machinery wearing
an interface coat. T-015 deletes the ceremony and expresses hot-reload as
what it is: atomic pointer swaps protected by CAS loops.

Hot-reload semantics are preserved exactly: configuration changes
propagate to in-flight operations via the pointer loaded at the top of
each method call, and to new operations via the resolver consulted on
entry. Race-detection tests exercise the swap path concurrently with
method calls for every new concrete type.

## What changed

### New packages

- **`internal/shared/objectstorage/`** — concrete `Client` with
  `atomic.Pointer[state]` holding active + fallback backends.
  Consumers now depend on `objectstorage.Backend` (a narrow interface
  with the same method surface as the deleted port) so tests substitute
  stubs without a shared-kernel mock package.
- **`internal/discovery/extractionpoller/`** — concrete `Poller` with
  `atomic.Pointer[state]` holding the active `Runner`. `Runner` is a
  narrow interface implemented by `discoveryWorker.ExtractionPoller`
  via method match; `RunnerFactory` is a bootstrap-provided closure
  that builds Runners on config change so the package does not import
  `discovery/services/worker` (avoids the integration-test import cycle
  via `services/command`).
- **`internal/discovery/schemacache/`** — concrete `Cache` combining
  the old `providerBackedSchemaCache` (per-call tenant-scoped Redis
  lease) and `dynamicSchemaCache` (TTL hot-swap) into one type.
  `atomic.Pointer[state]` holds the current TTL; the provider lookup
  stays per-call because it's tenant-scoped.

### Deleted

- `internal/shared/ports/object_storage.go` — **interface only**
  (`ObjectStorageClient`). The shared value types (`UploadOption`,
  `UploadOptions`, `WithStorageClass`, `WithServerSideEncryption`) and
  sentinel errors (`ErrObjectStorageUnavailable`, `ErrObjectAlreadyExists`)
  remain in the file because they flow across contexts independent of
  the interface.
- `internal/shared/ports/mocks/object_storage_client_mock.go` — the
  gomock generated against the deleted port. Replaced by
  `internal/shared/objectstorage/mocks/backend_mock.go`.
- `internal/discovery/ports/extraction_poller.go` + `extraction_poller_test.go`
- `internal/discovery/ports/schema_cache.go` + `schema_cache_test.go`
- `internal/bootstrap/dynamic_object_storage.go` + `dynamic_object_storage_test.go`
- `internal/bootstrap/dynamic_discovery_runtime.go` + `dynamic_discovery_runtime_test.go`

## Hot-reload behaviour verified

- **Observable by in-flight operations:** atomic.Pointer.Load() at the
  top of every method means in-flight calls keep their loaded backend
  and subsequent calls see the new one after the CAS succeeds. No
  torn reads (race tests confirm). Matches the previous mutex-based
  dynamic wrapper's guarantees with no locking on the hot path.
- **Resolver-driven rebuild preserved:** the ObjectStorage `Client`'s
  `Resolver` and the extraction `Poller`'s `ConfigResolver` are
  invoked on every method call, same as the old dynamic wrapper. CAS
  is a no-op when the cache key / config is unchanged, so the hot path
  has zero allocations in steady state.
- **Runtime observability preserved:** the existing systemplane
  logging at wire-up time (`discovery: extraction poller wired into
  command use case (poll: X, timeout: Y)`) continues to fire. The
  per-call log spam from the old wrappers is gone because the new
  types don't log on every call — they were already not supposed to.
- **Fail-fast semantics preserved:** the `runtimeObserved` gate on the
  ObjectStorage client still returns `ErrObjectStorageUnavailable`
  when a real backend was installed and then explicitly cleared,
  instead of silently reverting to fallback.

## Migration notes for downstream

- Code that used `sharedPorts.ObjectStorageClient` as a parameter
  type should now use `objectstorage.Backend`. `*objectstorage.Client`
  satisfies this interface, so the bootstrap-provided client flows
  through unchanged.
- Code that used `ports.SchemaCache` / `ports.ExtractionJobPoller`
  should depend on the concrete types (`*schemacache.Cache` /
  `*extractionpoller.Poller`) or the local narrow interfaces in
  `syncer` / `query` packages.
- Mock generation: the gomock file at
  `internal/shared/objectstorage/mocks/backend_mock.go` regenerates
  via `mockgen -source=internal/shared/objectstorage/client.go -destination=internal/shared/objectstorage/mocks/backend_mock.go -package=mocks`.

## Files not modified (intentional)

- Other `dynamic_*` wrappers in bootstrap (per Keep List): these
  have distinct concerns (fetcher client has SSRF guards, lock
  manager wraps Redis lease lifecycle, etc.) and are NOT single-
  implementation wrappers wearing a port-interface coat. They stay.
- The `shared/ports/object_storage.go` file itself still contains
  non-port value types and sentinels. Rename was considered and
  rejected: the file name is stable, and renaming would force 17+
  import updates for no behavioural benefit.
